## 摘要

代码智能体正在迅速发展，但其调试过程变得日益困难。由于框架在处理复杂任务时需要编排并行的工具调用和多阶段工作流，这使得智能体的状态转换和错误传播变得难以观察。在这些运行过程中，早期的失误可能会使智能体陷入无效循环，甚至演变成根本性错误，形成隐藏的错误链，从而难以确定智能体是*何时*偏离轨道以及*为什么*偏离。现有的智能体追踪分析要么侧重于简单的交互，要么依赖小规模的人工检查，这限制了它们在实际编码工作流中的可扩展性和实用性。我们提出了 \system，这是一种追踪架构，它通过演进式提取器解析异构的运行产物，将完整的状态转换历史重建为具有持久内存的分层追踪树，并执行**故障起点定位**，以精确定位故障源头及其下游链条。为了进行系统评估，我们构建了 \bench，该基准测试源自四个广泛使用的代码智能体框架在多样化代码任务（如错误修复、重构和终端交互）上生成的大量*已执行*轨迹，并在阶段级和步骤级为故障定位提供监督。实验表明，\system 的表现显著优于直接提示和轻量级基准方法，且回放其诊断信号可以在相同预算下持续修复原本失败的运行。我们的代码和数据已公开（https://github.com/NJU-LINK/CodeTracer）。

## 引言

大语言模型（LLM）正越来越多地用于驱动代码智能体，这些智能体能自主地与软件仓库和开发环境进行交互 [NEURIPS2024_5a7c9475, ICLR2025_a4b6ad6b, pmlr-v235-wang24h, xia2025demystifying]。这些智能体通过执行长序列的异构操作——如搜索代码、读取文件、编辑实现、运行构建以及解读测试反馈——来解决复杂的工程任务，例如仓库级缺陷修复或系统配置 [ICLR2024_edac78c3, merrill2026terminalbench]。随着此类智能体能力的提升，其执行过程变得越来越长且越发复杂，这使得理解故障变得愈加困难。例如，当一个智能体运行失败时，通常很难弄清楚轨迹最初是在哪里出错的，或者哪些中间决策导致了最终的失败。

目前的评估实践对这些过程的可见性非常有限。大多数基准测试使用端到端指标（如通过率或补丁正确性）来总结智能体性能，将整个轨迹简化为单一的成功或失败标签 [ICLR2024_edac78c3, merrill2026terminalbench, xia2025demystifying]。现有的智能体轨迹分析要么依赖于粗粒度的结果级判断，要么依赖于对少量运行的人工检查，这使得它们难以扩展到具有长执行轨迹的现实软件工程工作负载中 [bouzenia2025trajectories, enconda2025]。

在这项工作中，我们引入了 **CODETRACER**，这是一个将异构智能体运行目录转换为结构化分层轨迹并自动识别轨迹中最早故障关键阶段的框架。该系统还可以产生可操作的调试信号，通过反思回放反馈给智能体，从而实现对早期错误的针对性恢复 [shinn2023reflexion, madaan2023selfrefine, chen2024selfdebug]。

为了支持系统性评估，我们构建了 **CODETRACE-BENCH**，这是一个包含步骤级标注的高质量代码智能体轨迹基准。该基准汇总了由多个广泛使用的智能体框架在不同软件工程工作负载下生成的轨迹，包括仓库级缺陷修复和长周期终端交互任务 [ICLR2024_edac78c3, NEURIPS2024_5a7c9475]。每个轨迹都标注了结构化的步骤元数据和故障关键标签，从而能够评估阶段级的定位能力和证据检索能力。

利用该基准，我们对多种模型骨干和智能体框架下的代码智能体行为进行了大规模实证研究。我们的分析揭示了几种系统性模式，包括：**证据到行动的间隙**（evidence-to-action gap），即智能体经常检索到相关信息，但未能将其转化为正确的状态改变操作；即使是强大的模型，其在不同轨迹间的**行动效率**也存在显著差异；以及**阶段依赖型错误模式**，即故障关键决策往往集中在工作流的特定阶段。

总的来说，本工作的贡献如下：
1. **CODETRACER**：一个用于代码智能体执行中分层轨迹追踪和故障起点定位的可扩展框架。
2. **CODETRACE-BENCH**：一个包含数千条标注的代码智能体轨迹基准，能够实现过程级的系统性诊断评估。
3. 对现代代码智能体故障模式的大规模实证分析，揭示了关于其推理和执行行为的可操作见解。

## 引言

由大语言模型（LLM）驱动的代码智能体在真实代码仓库上编排长且异构的执行过程——交替进行搜索、阅读、编辑、构建和测试——其中早期的错误决策可能会触发连锁反应式的失败 [pmlr-v235-wang24h, NEURIPS2024_5a7c9475, ICLR2025_a4b6ad6b, xia2025demystifying]。当运行失败时，通常不清楚它最初在*何时*脱轨以及*为什么*脱轨。然而，目前的评估协议仍将整个运行过程简化为二进制的成功/失败标签，丢弃了丰富的过程信号；且先前的分析大多依赖于结果层面的评判或小规模的人工检查，在现实的预算限制下，无法扩展到步骤级、感知过程的诊断 [ICLR2024_edac78c3, merrill2026terminalbench, xia2025demystifying, bouzenia2025trajectories]。

我们引入了 \system，这是一个追踪框架，它将异构的运行目录标准化为分层追踪（hierarchical traces），并执行**失败起点定位**——即识别最早的失败关键阶段并检索紧凑的、基于证据的支持信息，同时可选地通过反思性重放（reflective replay）产生可操作的调试信号 [shinn2023reflexion, madaan2023selfrefine, chen2024selfdebug]。我们构建了 \bench，这是一个具有阶段级和步骤级监督的执行轨迹基准测试，并证明 \system 在定位准确性上优于基准方法，而由其输出引导的重放能够持续提升端到端的任务成功率。[fig:overview] 展示了我们的流水线。

**\system 流程概览。** 原始轨迹被标准化为分层追踪，整理成具有步骤级监督的 \bench，并通过失败起点定位（配合可选的反思性重放）进行诊断。

## 轨迹分析

### 轨迹采集
我们从评估代码生成和软件工程能力的基准测试中广泛采集了执行轨迹，涵盖了仓库级漏洞修复（*SWE-bench Verified*、*SWE-bench Pro*、*MultiSWE-bench*、*SWE-PolyBench*）和长程终端交互（*TerminalBench*）[NEURIPS2024_5a7c9475, ICLR2024_edac78c3, ICLR2025_a4b6ad6b, merrill2026terminalbench]。
每个基准测试在四种广泛使用的智能体框架（*SWE-Agent*、*MiniSWE-Agent*、*OpenHands*、*Terminus 2*）[NEURIPS2024_5a7c9475, ICLR2025_a4b6ad6b, merrill2026terminalbench] 下执行，并配对五个模型后端（*Claude-sonnet-4*、*GPT-5*、*DeepSeek-V3.2*、*Qwen3-Coder-480B*、*Kimi-K2-Instruct*）[deepseekai2025deepseekv32pushingfrontieropen, anthropic2025sonnet, kimiteam2026kimik2openagentic, qwen_qwen3_coder_next_tech_report, openai2025gpt5]，从而产生了一个跨越不同任务状态、编排策略和后端能力的多元化语料库。

### 过滤
为了在异构框架之间获得一致的进展单位，我们将每次运行规范化为由执行命令定义的*迭代*（iterations），并在此规范化空间中应用基于质量的过滤。
从 7,936 条原始轨迹开始，我们应用了四个连续的过滤器：
1. 移除在完成前超时的运行（保留 6,511 条）；
2. 移除生成追踪不完整或被截断的运行（6,109 条）；
3. 移除 Docker 环境配置错误或任务文件损坏导致执行输出不可靠的运行（5,284 条）；
4. 移除规范化步数少于 10 步的*正确*轨迹，这些轨迹通常是平凡完成的，为失败分析提供的信号有限（3,326 条）。
过滤后，我们保留了 3,326 条涵盖所有基准测试-框架-后端组合的轨迹。

### 标注

所有标注工作均由本文作者完成。
每位标注者被分配一组*任务*以及完整后端-智能体矩阵中对应的*所有*轨迹（每组任务 15 个组），并获得任务说明、参考解决方案以及访问执行环境的权限，以便对模糊案例（例如环境或配置问题）进行人工验证。
这种分配方式使每位标注者能够对任务建立深度理解，促进对不同模型和智能体处理同一问题方式的横向比较，并实现对相似失败模式的高效去重。

对于每一条轨迹，标注者为每一步分配一个*阶段标签*，指示工作流的阶段（环境验证、依赖安装、检查/调试、补丁修复、验证）。
对于*成功*的轨迹（即通过所有测试的轨迹），标注者进一步识别*冗余*和*试错*步骤——即对最终解决方案没有贡献或随后被撤销的操作。
对于*失败*的轨迹，我们采用了*基于链的反向追踪*协议：从失败的测试输出开始，标注者识别紧接在前的、其输出或操作导致了观察到的错误的步骤，并递归地向上游追踪，询问是什么早期的决策导致了这一中间失败，直到（i）前面的步骤不包含错误，或（ii）失败原因与早期的轨迹决策无关。
每条链终止于一个*错误关键步骤*（error critical step）——即触发下游级联反应的最早决策 [letsverify2023, processbench2025, enconda2025]——并从受控词汇表中获得一个*错误类型标签*（环境/设置问题、依赖解析失败、定位错误的编辑、错误的假设、验证误读以及无效循环）。
为了评估标注的可靠性，随机抽取了 15% 的轨迹子集进行独立的双重标注；标注者之间关于错误关键步骤标签的一致性达到了 Cohen's $\kappa = 0.73$。
完整的标注指南（包括边缘情况处理）见 [sec:app:annotation]。

标注过程揭示了跨后端和智能体框架的几种反复出现的模式，这些模式启发了 **system** 的设计。

**模型偏好与共享失败模式**
我们分析了所有 15 个后端-智能体组在各类别下的通过率（[fig:model_venn]）。
在 340 个任务类别中，所有五个模型都能解决的 66 个类别大多是常规的转换和脚本工作负载，如正则表达式处理、JSON/CSV 处理和标准数值程序。
在另一个极端，五个模型都无法解决的 65 个类别则偏向于需要更强的外部落地能力或更长程推理的任务，包括形式验证、计算机视觉、高级科学计算和过时环境。
在这两极之间，最强的后端保持着广泛的竞争力，但表现出明显的任务偏好：GPT-5 在图论密集型、化学和取证任务上相对较强；Claude-sonnet-4 擅长贝叶斯推理和投机采样；Kimi-K2-Instruct 在图形学和光线追踪方面表现突出；而 DeepSeek-V3.2 在数据流水线和包管理工作负载上更具优势。
当所有模型都无法真正解决任务时，它们的行为变得惊人地相似：它们往往不显式地承认失败，而是通过捏造证据、将占位符输出作为真实结果呈现，或在陷入无效循环后过早停止来绕过瓶颈。

**过度工程：当复杂性并不代表成功**
我们将轻量级智能体设计（*MiniSWE-Agent*）与逐渐复杂的框架（*Terminus 2*、*SWE-Agent*、*OpenHands*）在两个维度上进行了比较：端到端任务成功率和*过程成本*（由步数和 token 消耗衡量，见 [tab:arch_cost_success]）。相对于 MiniSWE-Agent，Terminus 2 增加了适度的编排开销（51.3k vs. 44.6k tokens），而 SWE-Agent 和 OpenHands 的 token 使用量几乎翻倍（86.7k 和 91.4k）。然而，这些额外的支出仅带来了微弱的成功率增益（+2.4-5.5 个百分点）。这种模式表明，对于通用的基于终端的代码任务，成功率主要受限于后端能力，而架构的复杂性往往转化为更高的成本和更长的交互循环，而非可靠的更优结果。

**要点 2：关于过度工程**
额外的编排增加了成本，但没有带来成比例的成功率提升；后端能力是主要的影响因素（[tab:arch_cost_success]）。

**阶段动态、失败起始与错误类型演变**
利用标注过程中分配的阶段标签（环境验证、依赖安装、检查/调试、补丁修复和验证），我们研究了阶段占比和转换与端到端成功率的关系。成功的运行表现出更连贯、向前推进的进展，往返振荡较少。相比之下，未成功的运行在早期设置活动和反复检查循环中分配了不成比例的预算，这通常发生在早期错误的承诺之后，随后的步骤无法撤销该错误。错误类型也跨阶段地发生系统性转变：环境和依赖错误集中在早期，而定位错误的编辑、错误的假设和验证误读在后期的补丁修复和验证中占据主导地位。错误关键步骤集中在少数阶段（[fig:error_stage_by_stage]），这直接激发了我们失败起点定位的目标。

**要点 3：关于阶段相关错误**
错误类型很大程度上由工作流阶段决定：环境和依赖错误主导早期阶段，而定位错误的编辑和错误的假设集中在后期的补丁修复和验证中。这种可预测性使得阶段感知防护栏能够抢在失败级联之前将其制止，而不是仅仅依赖于事后诊断（[fig:error_stage_by_stage]）。

**成功率随迭代次数增加而趋于饱和**

我们将 `max_iterations` 在 $\{5, \dots, 300\}$ 范围内进行遍历，并在每种预算下重新评估每对“骨干网络—框架”组合。[表 tab:success_vs_iterations_3agent_merged] 报告了在骨干网络层面汇总的问题解决率和 Token 使用量，而每种组合完整的 $5 \times 3$ 轨迹详见 [图 fig:iter_grid_app]。研究结果呈现出两个一致的模式。第一，随着迭代预算增加到 40 步左右，成功率迅速提高，但此后收益迅速减少，曲线基本趋于平缓。第二，“*饱和天花板*”主要取决于骨干网络：更强大的模型在更高的问题解决率处进入平台期，但达到平台期的时间点并无显著推迟。这表明，额外的迭代主要有助于从低预算下的探索不足中恢复，而不是从根本上提高智能体的推理能力。当智能体在早期就认定了一个错误的假设时，额外的迭代通常被消耗在冗余的探索和反复试错的编辑上，而不是纠正根本性的错误。

**要点 4：关于迭代预算**
成功率迅速饱和；额外的迭代会放大无效循环，而无法实现恢复（[表 tab:success_vs_iterations_3agent_merged]）。

## CodeTracer

### 方法

CodeTracer 致力于解决**故障发生定位**（failure onset localization）问题：给定一个被分割为有序阶段的标准轨迹 $\tau$，它预测一个导致故障的阶段 $\hat{s}$，该阶段内的一组与错误相关的步骤 $P$，以及支持该诊断的精简证据集 $E$。监督信息根据我们的标注指南标记在因果上导致最终故障的阶段，而评估则是在匹配的预算下，针对黄金标准错误步骤集 $G$ 衡量步骤层面的精确率/召回率/F1 值以及 Token 成本。

由于运行目录是异构的，且其文件布局依赖于智能体，硬编码的解析器往往非常脆弱。因此，CodeTracer 将**演进式提取**、**树索引**和**诊断**分离，同时积累可复用的解析器，使得对新格式的支持随时间推移而不断改进，这顺应了软件工程智能体中对于过程感知轨迹分析的更广泛需求 [bouzenia2025trajectories, enconda2025]。

**流水线**。CodeTracer 将追踪过程分解为三个阶段（详见 [sec:app:framework]）：
1. **演进式提取**：扫描运行目录，生成描述哪些工件记录了执行步骤的精简布局规范，并据此调整解析器：它首先检查现有解析器的注册表；若未发现匹配项，则合成并注册一个新解析器，然后使用固定的查询模式实例化特定格式的诊断提示。该阶段输出带有类型化字段（动作、观察、差异、验证结果）的归一化步骤记录，同时防止跨运行的格式偏移。
2. **树索引**：将归一化步骤的扁平序列转换为**分层结构的**追踪树 [fig:trace_format_example]。仅检查当前环境而不修改代码库或执行状态的步骤被表示为同一状态下的探索节点，而修改代码库或执行环境的步骤则被表示为状态改变节点，引导至子状态的转换。每个节点都进一步标注了意图和结果的摘要。生成的树明确了后续动作是在未改变的上下文中执行的还是在干预后执行的，并作为诊断的压缩导航索引。
3. **诊断**：遍历该树，发出结构化的证据查询，并输出导致故障的阶段、与错误相关的步骤以及证明定位合理性的精简证据集。

分层追踪树。探索步骤保留在当前状态节点下，而状态改变步骤则引发向子状态的转换。CodeTracer 在树索引期间构建此结构，以提供执行追踪的压缩导航。

## CodeTraceBench

### CodeTraceBench

我们直接从 [sec:study] 中的轨迹语料库构建了 CodeTraceBench。该来源池涵盖了五个基准测试（SWE-bench Verified、SWE-bench Pro、MultiSWE-bench、SWE-PolyBench 和 TerminalBench）、五个骨干模型（Claude-sonnet-4、GPT-5、Kimi-K2-Instruct、Qwen3-Coder-480B、DeepSeek-V3.2）以及四个框架（SWE-Agent、MiniSWE-Agent、OpenHands、Terminus 2），总计 4,354 条标准化且具有步骤级标注的轨迹 [ICLR2024_edac78c3, NEURIPS2024_5a7c9475, ICLR2025_a4b6ad6b]。我们从该池中精选了具有清晰失效级联（failure cascades）和充足追踪内证据的长跨度实例，同时移除了过短或近乎重复的运行记录。

## 实验

### 实验设置

我们使用交集子集（具有完全覆盖范围的基座-方法对）来评估 CodeTraceBench。我们测试了 Claude-sonnet-4、GPT-5 和 DeepSeek-V3.2 在三种定位方法（原始 LLM、\miniSystem、\system）下的表现，所有方法都在相同的运行目录下操作，并匹配了预算和统一的解码设置。我们报告了宏观平均步骤级精确率（Precision）、召回率（Recall）和 F1 分数（正式定义见 [sec:app:metrics]）以及总 Token 使用量。

### 主要结果

[tab:tracebench_main] 总结了主要的定位结果。在所有基座模型中，与原始日志提示相比，添加结构和追踪信号显著提高了定位质量。\miniSystem 已经恢复了部分增益，表明轻量级标准化是一个强大的基线，而 \system 通过优先处理真正与失败相关的步骤而非仅仅是显著的人工痕迹，进一步提高了精确率和召回率。Token 效率也得到了提升，因为追踪将证据检索缩小到紧凑的候选集中。

三个前沿基座模型达到了相当的 F1 分数（46--48%），但在遍历追踪的方式上有所不同。GPT-5 较早地终止了搜索，以高置信度致力于一组紧凑的失败相关步骤；这产生了最佳的精确率（总体 45.0%，简单任务 49.9%）和最低的 Token 成本（总体 31.1k，简单任务 20.6k），但遗漏了一些导致错误的步骤。Claude-sonnet-4 在得出结论前继续深入扫描追踪轨迹，发现了更多与失败相关的证据，并在所有划分中实现了最高的召回率（总体 54.9%，简单任务 59.1%），代价是更高的 Token 消耗（56.8k）和较低的精确率（40.5%）。DeepSeek-V3.2 介于两者之间，在覆盖范围和简洁性之间取得了平衡，其在不同难度级别下的 P/R 差距最为均匀。

更难的任务与更长的轨迹以及比例更高的 Token 预算相关——每个基座模型的简单到困难 Token 比例大约翻了一番——这证实了定位难度随着轨迹复杂性的增加而扩展。我们进一步按难度、类别和执行阶段（附录）对结果进行了分层；各方法之间的相对排序保持稳定，其中 \system 在失败相关证据分散的较难实例上提供了最大的收益。

**诊断行为的要点 5**：前沿基座模型实现了相似的 F1，但表现出不同的搜索深度策略：GPT-5 提前结束以较低成本获得更高精确率，Claude-sonnet-4 进行详尽搜索以获得更高召回率，而 DeepSeek-V3.2 平衡了两者。定位成本随轨迹长度扩展，而非模型身份 [tab:tracebench_main]。

### 分析与回放

我们利用 \system 的层次化追踪结构和诊断输出进行分析。

**证据与动作的差距。** [fig:evidence_action_gap] 将每个轨迹的步骤预算分解为正确的状态变化、有用的探索和无效步骤。在所有五个模型中，无效部分的比例从已解决（22%）到未解决（40%）几乎翻了一番，而正确的状态变化则持续下降（30% $\to$ 21%）。探索的有效性仅略微下降，这表明智能体仍然收集了相关信息，但未能将其转化为正确的动作——这是一个理解瓶颈，其严重程度随基座模型而异（Qwen3-Coder-480B 和 Kimi-K2-Instruct 的下降最为剧烈，分别为 $\Delta=11.7$ 和 $10.3$ 个百分点）。

**证据与动作差距的要点 6**：智能体通常通过探索收集有用证据，但未能将其转化为有效的状态变化动作 [fig:evidence_action_gap]。

**组件消融。** 我们在 \miniSystem 之上增量添加了 \system 的两个关键组件——演进式提取（布局发现 + 解析器适配）和树索引 [tab:ablation]。树索引通过提供压缩的层次化导航带来了最大的单项增益（18.3 点 F1），而演进式提取通过解析器重用的格式标准化贡献了额外的 9.4 点提升。

**反思式回放。** 我们将 \system 定位的证据反馈给智能体：在最初失败的运行中，相同的基座模型在匹配的预算下被再次调用，并将诊断结果作为前缀提示注入 [fig:replay] [shinn2023reflexion, madaan2023selfrefine, chen2024selfdebug]。回放一致地提高了所有基座模型的 Pass@1，帮助智能体修改早期的错误决策，而不是重复无益的探索。诊断过程本身对于 Claude-sonnet-4 平均消耗 8.4k Token，GPT-5 为 5.2k，DeepSeek-V3.2 为 7.1k；这些 Token 在回放预算 *之外* 计算，以便回放运行获得与原始运行完全相同的迭代和 Token 预算。

**工业级智能体分析。** 除了上述研究的学术智能体框架外，我们还应用 \system 分析了来自 Claude Code（一个工业级编程智能体）的轨迹。与研究型智能体相比，Claude Code 采用了丰富得多的工具基础设施（涵盖 8 个类别的 40 多个专用工具）和复杂的上下文管理（压缩、Token 预算），产生了较低的探索与变化比例，这与更高的轨迹效率相关；然而，工业智能体特有的并行工具执行引入了顺序敏感性问题，这在顺序执行的学术框架中是不存在的。完整的架构比较和轨迹统计见 [sec:app:claude_code]。

图片：**反思式回放。** 在匹配预算下，注入 \system 的诊断信号前后，最初失败运行的 Pass@1 结果。

**动作效率。** 我们将 *有效动作比率* 定义为每个轨迹中推进目标的步骤所占的比例。[fig:effective_action_ratio] 显示，更强大的模型实现了更高的平均值（Claude-sonnet-4 为 73%，GPT-5 为 71%），但都表现出广泛的分布且具有明显的左尾——每个基座模型都包含相当比例的轨迹，其中只有不到一半的动作是在推进目标的。

## 相关工作

**代码智能体与软件工程基准测试。**
近期研究推进了在真实开发环境中的软件工程智能体，包括仓库级漏洞修复和长周期终端交互。代表性的系统和范式包括 SWE-Agent、OpenHands、CodeAct 风格的可执行动作智能体以及 Agentless，而评估则越来越多地集中在可执行基准测试上，如 SWE-bench 和 Terminal-Bench [NEURIPS2024_5a7c9475, ICLR2025_a4b6ad6b, pmlr-v235-wang24h, xia2025demystifying, ICLR2024_edac78c3, merrill2026terminalbench]。然而，这些工作主要强调端到端的任务成功率、补丁正确性和解决率，对于长智能体轨迹在何处首次变得对失败至关重要，所提供的可见性有限。

**轨迹错误定位、调试与重放。**
最近的研究也开始探讨软件工程智能体的轨迹和过程级质量 [bouzenia2025trajectories, enconda2025]。其他领域的相关基准测试工作进一步表明，步骤级监督和过程错误识别可以揭示仅凭结果评估会遗漏的失败 [letsverify2023, processbench2025]。我们的定位目标也与问题定位、经典故障定位以及基于大语言模型（LLM）的调试相关：以往的工作已经为软件工程任务定位了与问题相关的代码区域，从失败的执行中对可疑程序元素进行排序，并利用运行时追踪来调试候选程序 [xia2025demystifying, jones2005tarantula, abreu2009sfl, zhong2024debuglikehuman, chen2024selfdebug]。基于反思的自我改进方法（如 Reflexion 和 Self-Refine）进一步展示了批评、反馈和记忆如何改进后续尝试 [shinn2023reflexion, madaan2023selfrefine]。相比之下，我们专注于执行后的智能体轨迹上的分层追踪标准化、阶段级失败起点定位、证据检索以及面向重放的诊断输出。

为了完整性和可复现性，附录总结了完整的标注协议、额外的框架和指标细节，以及用于诊断和重放的完整提示词和输出模式，并提供了扩展的实证结果，包括跨难度设置和迭代预算的各骨干模型和各框架的详细细分，以支持正文的发现。

## 结论

我们提出了 **CodeTracer**，这是一个将异构代码智能体日志转换为结构化分层追踪并执行自动化故障起点定位的追踪框架。同时，我们还推出了 **CodeTraceBench**，这是一个包含数千条步骤级标注轨迹的基准测试，涵盖了四个智能体框架、五个前沿骨干模型以及多种软件工程任务。我们对该语料库的实证研究表明：智能体经常能收集到正确的诊断证据，但未能据此采取行动；一旦达到骨干模型的推理能力上限，额外的编排复杂度和迭代预算所带来的收益就会递减；且错误类型在工作流的不同阶段呈现出可预测的偏移。在步骤级定位方面，**CodeTracer** 显著优于原始日志提示（raw log prompting）和轻量级基准方法，在更低的 Token 成本下达到了高达 48% 的宏观 F1 值。此外，在相同预算下，通过反思式重放（reflective replay）注入其诊断输出，能够稳定地修复最初失败的运行任务。

## 局限性

本研究存在若干局限性。首先，尽管 CodeTraceBench 是由跨多个智能体框架、主干模型和工作负载的执行轨迹构建的，但它并未涵盖软件工程智能体或真实世界代码仓库的所有设计空间；因此，关于失败模式和回放增益的结论可能无法均匀地推广到其他框架、领域或部署设置中。其次，我们的监督依赖于对失败关键行为的阶段级和步骤级标注。虽然我们的指南旨在使这些判断保持一致，但诸如“不正确”、“无用”和“错误关键”等标签仍涉及标注者的主观解读，特别是对于探索与状态变化相互交织的长轨迹。第三，我们的追踪和回放结果是在先前收集的运行记录上，基于匹配的离线预算进行评估的。这种设置隔离了诊断质量，但无法完全捕捉智能体如何适应交互式的人类监督、变化的环境或重复的在线干预。最后，我们的反思式回放实验测试了局部证据是否能帮助修复失败的运行，但其本身并未建立通用的训练信号，也不能保证在所有任务类别和模型系列中实现稳健的提升。因此，我们将 \system 和 \bench 视为结构化诊断和受控评估的工具，而非对代码智能体可靠性的完整阐述。

## A. 标注指南（对应第 2.3 节）

## 标注指南

本附录提供了 [sec:annotation] 中描述的标注架构的详细定义。每位标注人员都会收到任务规范、参考解决方案以及执行环境的访问权限，并被分配处理每个任务的所有 15 条“基座模型-智能体”轨迹。

**阶段标注（所有轨迹）。**
轨迹中的每一步都被分配一个来自以下有序词表的*阶段标签*：
(i) *环境验证* —— 确认运行时、工具链和测试套件；
(ii) *依赖安装* —— 安装、升级或固定软件包和系统库；
(iii) *检查 / 调试* —— 阅读源文件、搜索代码、运行测试、检查日志；
(iv) *补丁编写* —— 修改源代码、配置文件或构建脚本；
(v) *验证* —— 重新运行测试、代码检查（linting）或其他用于确认补丁的验证工作。
一条轨迹可能会重新访问某些阶段（例如，在补丁失败后返回检查阶段）；在这种情况下，每个连续的块都会被分配其专属的阶段跨度。

**成功的轨迹。**
对于通过所有测试的轨迹，标注人员需识别两类无贡献步骤：
(i) *冗余*步骤：其效果完全被早期步骤涵盖的动作（例如，重复阅读已检查过的文件，重新应用相同的补丁）。
(ii) *试错*步骤：随后被撤销或取代的动作（例如，一个错误的补丁尝试，随后被正确的修复所覆盖）。

**失败的轨迹 —— 基于链的回溯追踪。**
对于失败的轨迹，标注人员根据失败的测试输出进行递归回溯追踪：
(i) 识别紧邻的前一步，其输出或动作产生了观察到的错误；
(ii) 通过询问哪项早期决策导致了这一中间失败来向上游追踪；
(iii) 重复此过程，直到前序步骤不包含错误，或者失败原因与早期的轨迹决策无关。
该追踪链终止于一个*错误关键步骤* —— 即触发下游级联反应的最早决策。
每个错误关键步骤都会被分配一个来自受控词表的*错误类型标签*：
环境/设置问题、依赖解析失败、错误定位的编辑、错误的假设、验证误读以及无成效的循环。

**边缘情况。**
(i) 如果某个步骤具有歧义（例如，部分正确的补丁），标注人员默认将其标记为*不正确*并注明歧义。
(ii) 由于外部因素（网络超时、软件包缺失）而失败的环境设置步骤，仅在智能体选择了本可以避免的路径时才被标记为*不正确*；标注人员通过进入执行环境进行核实。
(iii) 当多个阶段包含不正确步骤时，包含错误关键步骤的*最早*阶段被指定为导致失败的责任阶段。

## 追踪框架详情

\system 的三阶段流水线在 [sec:ta_tracer_system] 中进行了总结。下文我们将提供关于评分特征、重放协议和计算复杂度的更多细节。

**评分特征。**
追踪代理使用以下信号对候选阶段进行排序：
(i) *验证回归*：某个阶段的状态变更步骤是否导致之前通过的测试失败；
(ii) *差异幅度*：该阶段内状态变更步骤中累计修改的行数；
(iii) *回溯频率*：后续阶段撤销或重新尝试该阶段工作的次数；
(iv) *探索与行动比率*：该阶段中状态变更步骤与探索性步骤的比例。

**重放协议。**
在生成用于代理改进的调试信号时，\system 执行*反思性重放*：它将诊断出的失败关键阶段连同提取的证据提交给骨架模型，并要求模型提出纠正措施。重放受到预算限制（上限为与原始运行匹配的固定 Token 限制），以确保与无辅助基准线的公平比较。

**复杂度。**
演进式提取（布局发现和解析器选择）针对每种唯一的框架格式调用一次，并分摊到共享该格式的所有运行中。单次运行的成本主要由诊断过程决定，该过程随阶段数量（通常每个轨迹 3--15 个）线性扩展。\system 的总 Token 开销在 [tab:tracebench_main] 中报告。

## 评估指标

对于每次运行，令 $G$ 表示 CodeTraceBench 中标注的人工金标准故障相关步骤集合，$P$ 表示预测集合。（如果某种方法生成了排序列表，我们将所有返回的候选结果视为 $P$；如有必要，我们会根据各方法共用的固定实现预算对 $P$ 的数量进行上限限制。）

**召回率 (Recall)** 衡量金标准证据的覆盖范围：
$$\mathrm{Rec} = |P \cap G| / |G|$$

**精确率 (Precision)** 衡量预测证据的选择性：
$$\mathrm{Prec} = |P \cap G| / |P|$$

**F1 分数** 综合平衡上述两项指标：
$$\mathrm{F1} = 2\cdot \mathrm{Prec}\cdot \mathrm{Rec} / (\mathrm{Prec}+\mathrm{Rec})$$

**指标聚合**
我们通过计算每个实例的指标并取平均值来报告*宏观 (macro)* 精确率/召回率/F1 分数，从而确保无论 $|P|$ 或 $|G|$ 的大小如何，每条轨迹的贡献都是平等的。此外，我们还报告了 LLM 的总 Token 数，作为一种效率衡量标准。

## 补充实验结果

**按类别细分**
我们根据分类法中定义的 26 个任务类别对 CodeTraceBench 的定位结果进行了分层。各方法之间的相对排序（基础大模型 $<$ \miniSystem $<$ \system）在各类别中保持一致，其中 \system 在具有长周期、多阶段失败级联的类别中（例如：*构建系统*、*依赖解析*、*多文件重构*）表现出最显著的提升。具有短周期、单阶段轨迹的类别（例如：*简单修复*、*配置编辑*）表现出较小的绝对差异，因为失败的关键阶段通常很容易识别。

**难度缩放**
我们根据轨迹长度将实例分组（短：$\leq$15 步，中：16--40 步，长：$>$40 步）作为难度的替代指标。\system 相对于 \miniSystem 的 F1 优势随轨迹长度增加而扩大：在短轨迹上提升了 1.2 个百分点（pp），中等轨迹提升了 3.8 pp，长轨迹则提升了 5.1 pp。这证实了当失败证据分散在多个步骤中时，结构化追踪的价值最大。

**其余基座模型**
两个额外基座模型（Kimi-K2-Instruct, Qwen3-Coder-480B）的结果与三个主要基座模型的趋势一致。Kimi-K2-Instruct 在所有方法中的整体 F1 略低，这与其在轨迹分析中观察到的较低有效动作率一致。Qwen3-Coder-480B 在定位任务上的表现与 DeepSeek-V3.2 相当。

## 工业智能体分析：Claude Code

我们应用 [system] 分析了工业编程智能体 Claude Code 生成的轨迹，并将其架构和行为与正文中研究的学术智能体框架进行了比较。

**架构。**
Claude Code 的源代码包含 40 多个专门工具，分为八个类别（文件操作、Shell 执行、搜索与导航、智能体编排与规划、Web 及外部服务、工作空间配置、任务管理等），而典型的学术智能体仅有 5–10 个工具。该系统还包括用于上下文压缩、Token 预算管理和功能门控代码路径的专用模块，这些在研究框架中并不存在。

**主要发现。**
我们的分析揭示了工业智能体与学术智能体之间的几点结构性差异：
1. **工具投入。** 工业智能体在专用工具和错误恢复基础设施上投入巨大，而学术智能体则使用狭窄的通用工具集。
2. **上下文管理。** 生产级智能体实现了复杂的上下文管理（压缩、预算跟踪、功能门控），这些功能在学术智能体中通常是缺失的，从而支持了更长的有效轨迹。
3. **探索与变更比率。** 探索与变更比率是轨迹质量的一个强预测指标：Claude Code 表现出较低的比率（每个探索步骤对应更多操作），这与更高的任务成功率相关。
4. **并行执行。** 工业智能体支持的并行工具执行显著减少了实际运行时间，但引入了顺序敏感性问题，这在顺序执行的学术框架中是不存在的。
5. **强化学习（RL）反馈信号。** [system] 在工业智能体轨迹上产生的每步偏差标签可以作为稠密训练信号，有可能弥补工业与学术智能体之间的行为差距。

需要注意的是，由于 Claude Code 的轨迹是在不同的任务分布上收集的（通过 Claude Code CLI 执行的 TerminalBench 任务），因此与正文中学术智能体结果的直接数值比较并不直观。因此，上述发现重点关注定性的架构和行为差异，而非聚合指标的对比。
